Theme Review

必須事項

テーマが WordPress.org の公式レポジトリに掲載されるためには、以下の必須事項すべてを満たしていなければなりません。 これらのチェックをする際は、テーマチェックプラグインの利用も必ずしてください。自動で必須項目のチェックをしてくれます。

アクセシビリティ

  • テーマに 'accessibility-ready' がついている場合には、こちらの必須事項も満たしている必要があります。

コード

  • PHP、JSのエラーがないこと。
  • 少なくともindex.phpとstyle.cssが必ずあること。
  • 有効なDOCTYPE宣言がされていること、language_attributesが含まれていること。
  • すべてをサニタイズする。
  • non-presentational hooksを削除したり変更したりしないこと。
  • ショートコードを使ってはいけない。
  • WordPress が生成する以下のcssのクラスをサポートしていること。
    • alignleft
    • alignright
    • wp-caption
    • wp-caption-text
    • gallery-caption
    • sticky （スタイリングされていなくてもOK）
    • bypostauthor （スタイリングされていなくてもOK）

コアの関数と機能Core Functionality and Features

  • WordPress の関数と機能を優先して使うこと。
  • admin/feature pointersを含めないこと。
  • カスタム投稿タイプとカスタム分類を使うことはできない。
  • お金を払わないと、WordPressの機能が使えなくなるようなペイウォールを設けてはいけない。
  • ツールバーを隠してはいけない。
  • （親）テーマへのパスを取得する際、TEMPLATEPATH定数ではなく、get_template_directory()を利用すること。
  • （子）テーマへのパスを取得する際、STYLESHEETPATH定数ではなく、get_stylesheet_directory() を利用すること。
  • コンテンツを変更するためにハードコーディングをしないこと。代替として、関数の引数や適切なフィルターやアクションを利用すること。たとえば、wp_titleはフィルターを使って変更されるべきです。
  • comments_template()が含まれていること。

ドキュメンテーション

  • 独自機能、オプション、制限（たとえばメニューに制限があるというような）がある場合には、きちんとした説明が必要です。十分なドキュメンテーションを提供しましょう。

ファビコン

  • ファビコンを利用するのであれば、デフォルトでは無効化しておき、ユーザーが上書きできるようにしておくこと。

言語

  • すべての文字列が翻訳対応していること。
  • style.css にテキストドメインが記述されていること。
  • テキストドメインには、単一のユニークな、style.cssに記述されたテーマスラッグを使うこと。フレームワークを利用する場合、テキストドメインは2つまで利用できます。
  • 文字列にはどの言語でも使えるが、すべての文字列がひとつの言語で記述されていること。

ライセンス

  • 100% GPL または、 100% GPL-compatilbe としてライセンスされていること。
  • 著作権とライセンスが明示的に宣言されていること。style.cssのヘッダーに license と license uri が使われていること。
  • フォントや画像など、すべてのリソースのライセンスが宣言されていること。
  • すべてのコードとデザインがあなたものであるか、法的にあなたに属したものであること。デザインのクローンは受け付けられません。

テーマ名

  • テーマの名前に、WordPress、Themeを利用することはできません。
  • "WordPress" のスペルを正確に。スペースを空けない、WとPが大文字です。

オプションと設定

  • オプションはひとつの配列に格納して保存すること。
  • 実質のあるデフォルトを利用すること。デフォルトの設定値をデータベースに保存しないこと。Use sane defaults and must not write default setting values to the database.（訳注：sane defaultの訳にぴったり来るのがないので英語残し）
  • add_theme_page() の権限には edit_theme_options を利用して、 administrator などのロールを使わない。また、 edit_themesや manage_options などの他の権限を利用しない。
  • オプションページを利用する場合には、「外観」の配下に設置すること。
  • テーマオプションを活用する場合には、 Settings API か Customizer を利用すること。
  • すべてのオプション名、独自の関数名、独自の変数名、独自の定数名には、テーマスラッグのプレフィックスをつけること。

プラグイン

  • プラグインを含まないこと。テーマ内でプラグインの利用をおすすめすることはできますが、テーマに含めることはできません。
  • プラグインテリトリーとして考えられていることをしないこと。

スクリーンショット

  • Screenshot.png は、ロゴやモックアップではなく、デフォルトの設定で有効化した時の、実際のテーマのものを使うこと。
  • screenshot.png は、880 x 660px よりも大きくしない。

セキュリティとプライバシー

  • ユーザーの同意なしに、データを送信しないこと（Don't phone home without informed user consent.）。追加情報は、セキュリティページに有ります。.
  • ユーザーのデータを収集する場合には、すべてオプトインとし、デフォルトで無効化されているオプションを設けること。
  • データベースに保存される信頼されないすべてのデータのバリデーションとサニタイズを実施していること。出力されるすべての信頼されないデータは、出力前にエスケープされていること。（Data Validationのページも見てください）
  • テキストフィールド内の値には esc_attr() を、テキストエリア内の値には esc_html() を利用すること。

販売、クレジット、リンク

  • テーマ販売をしている場合には、それらのテーマが GPL で販売されている必要があります。でなければ、 WordPress.org の公式レポジトリに（訳注：無料版の）テーマを掲載できません。
  • フッターにクレジットリンクを載せる場合には、ひとつだけ。

スタイルシートとスクリプト

  • スクリプト、スタイルシート、ファビコンのハードコーディングは、ブラウザ対策の場合を除き、ハードコーディングされてはならず、すべてenqueueされる必要があります。
  • 解析やトラッキングは許可されません。
  • ミニファイされたスクリプトを含める場合には、オリジナルのファイルも同梱しなければなりません。
  • スクリプトは、自分のバージョンを使うのではなく、コアにバンドルされたものを使わなければなりません。たとえばjQueryなど。
  • スクリプトとスタイルは、直リンクではなくテーマ内に同梱しなければなりません。例外はGoogleのライブラリです。
  • スタイルシートでタグが利用されている場合、その機能を有していなければならず、タグの意味するところをきちんと満たしていなければなりません。たとえば、カスタム背景やカスタムヘッダーなど。

テンプレート

  • テープレートをを利用する際、カスタムテンプレートは get_template_part() か locate_template() によって読み込むこと。
  • There are several template specific things you should consider when certain ones are being used.
  • *_url() テンプレートタグを使い、対応する bloginfo() 系のものを使わないこと。
以上の必須項目の多くについて自動化を進めていることを付記しておきます。
